# Roadmap v2 / v3 — après le tag v1.0.0

> Capture de la demande proprio (2026-05-31) : reprendre les modules C#/.NET, **ajouter un niveau de
> difficulté** aux exercices ET aux rushes, et couvrir de nouveaux sujets. La v2 est un gros morceau,
> découpée ici en **v2 (approfondissement C#/.NET, compatible moulinette actuelle + packaging modeste)**
> et **v3 (plateformes & architecture, nécessite un nouveau moteur/format)**.

## Légende format de notation
- `io` : exercice console déterministe (moteur actuel, OK).
- `io+pkg` : `io` mais nécessite d'embarquer un package dans le binaire (cf. décision packaging It.20).
- `lecture` : module guidé (cours + checklist), pas d'auto-notation (comme M05/M14/M22).
- `projet` : nécessite un nouveau type de grader (multi-fichiers / web / natif / DB).

## v2 — Approfondissement C#/.NET

### A. Niveau de difficulté (transverse) — DÉCISION moteur
Ajouter une notion de difficulté aux exercices et rushes. Pistes :
- Champ `difficulty: facile|moyen|difficile` dans `manifest.yaml` (affiché par `list`/`subject`).
- Exercices **bonus** plus durs (`ex03+`) marqués bonus, non bloquants dans la correction séquentielle.
- Variantes « difficile » des rushes.
→ Petit changement moteur (modèle manifest + affichage + éventuelle logique « bonus non bloquant »).

### B. Nouveaux modules `io` (moteur actuel suffit)
| Sujet | Format | Idées d'exos |
|---|---|---|
| **switch / pattern matching (C# 14)** | io | switch expression, patterns (type/relationnel/propriété/liste), `when` |
| **enums** | io | enum + `switch`, `[Flags]`, `Enum.Parse/ToString` |
| **static / const / readonly / immutable** | io | classe statique, `const` vs `static readonly`, `record`/`readonly struct`, immutabilité |
| **opérations binaires** | io | `& | ^ ~ << >>`, masques, parité de bits, conversions base 2 |
| **Big O / complexité** | io | implémenter et comparer (compter les opérations), raisonner sur la complexité |
| **algorithmes de tri** | io | bubble/insertion/quick/merge implémentés à la main, sortie déterministe |
| **A\* / recherche de chemin** | io | grille ASCII en entrée → chemin/longueur en sortie (déterministe) |
| **design patterns (suite)** | io | Decorator, Adapter, Command, Singleton, Builder (en plus de Strategy/Factory/Observer du M23) |
| **smelly code / refactoring** | io | identifier un code smell, livrer une version refactorée au comportement identique |
| **discriminated unions** | io | C# n'a pas d'union native → pattern hiérarchie scellée (`abstract record` + `sealed`), ou lib `OneOf` (→ `io+pkg`) |

### C. Modules `io+pkg` (packaging à décider)
| Sujet | Package | Note |
|---|---|---|
| **Entity Framework Core** | `Microsoft.EntityFrameworkCore.Sqlite` | SQLite **in-memory** → déterministe et auto-notable ; embarque le natif SQLite (quelques Mo) |
| **garbage collection** | (aucun) | surtout conceptuel : `GC.Collect`, générations, `IDisposable`/`using`, finaliseurs — io possible (compter, observer un dispose) ou `lecture` |
| **interopérabilité** | (aucun) | `LibraryImport`/`DllImport` vers libc/kernel32 — fragile/portabilité ; probablement quelques exos `io` simples + reste `lecture` |

### D. Débloquer depuis `v1.0-blockers`
- **M19 Logging** + **M20 Host** (résoudre la capture du provider logging) — cf. BLOCKERS-v1.0.md.
- **Grader « élève-écrit-tests »** pour valoriser M13.

## v3 — Plateformes & architecture (nouveau moteur/format)

| Sujet | Format | Ce qu'il faut |
|---|---|---|
| **Blazor (web)** | projet | harnais web (build + rendu/headless ou assertions DOM) ; sort du modèle console |
| **Silk.NET (rendu graphique)** | projet/lecture | fenêtrage/OpenGL natif, non déterministe headless → projet guidé ou capture d'image |
| **Docker** | lecture/projet | conteneurisation (le SDK .NET 10 sait publier sans Dockerfile, cf. skill) ; guidé, ou grader qui build une image |
| **Clean Architecture** | projet | structure multi-projets (Domain/Application/Infra/Api) → **grader multi-fichiers/solution**, + un rush dédié |
| **Grader git dédié** (M05/M14) | moteur | inspection LibGit2Sharp (cf. BLOCKERS-v1.0.md) |
| **Harnais réseau** (M22) | moteur | serveur de test embarqué (cf. BLOCKERS-v1.0.md) |
| **Rush 3** (Worker complet) | io+pkg | après résolution logging/host |

→ v3 implique un **grader « projet »** (compiler/exécuter une solution multi-fichiers, voire un host web)
au-delà du grader `io` mono-fichier actuel. C'est le gros chantier moteur.
