# Sprint 2 (V3) — Rush 3 « Traitement de commandes » (issue #2)

> Scrum / loop V3. Branche `feat/rush-03`. Contenu pur (moteur `io` existant, logging/host débloqués).

## Objectif
Rush de synthèse **Worker Service déterministe** : un `BackgroundService` single-shot consomme une
**file en mémoire (`Channel<T>`)** — qui *simule* une arrivée réseau — de commandes injectées par
**DI**, journalise chaque décision via le provider `LogCapture` fourni (déterministe), produit un
bilan, puis s'arrête (`IHostApplicationLifetime.StopApplication()`). La partie réseau réelle reste
pratique locale non notée (cf. BLOCKERS Point 3).

## Spec
Entrée : un entier `N`, puis `N` lignes `nom montant` (montant entier, séparé par un espace).
- Producteur : écrit les `N` commandes dans un `Channel.CreateUnbounded<Commande>()`, `Complete()`.
- Consommateur (`ReadAllAsync`, FIFO déterministe) : `Validateur` (service DI) accepte si `montant > 0`.
  - acceptée → `LogInformation("Commande acceptée : {Nom} ({Montant})")`, cumule total + compteur.
  - rejetée → `LogWarning("Commande rejetée : {Nom}")`, compteur.
- Fin → `LogInformation("Bilan : {A} acceptée(s), {R} rejetée(s), total {T}")` puis `StopApplication`.

Sortie (via `LogCapture`, catégorie = nom du worker `Traitement`) :
`Traitement [Information] ...` / `Traitement [Warning] ...`.

Livrables : `Traitement.cs` (Program top-level + types) + `LogCapture.cs` (FOURNI).
