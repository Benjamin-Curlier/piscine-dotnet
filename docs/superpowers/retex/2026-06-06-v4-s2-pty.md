## Décisions techniques

- **Lib PTY = `Porta.Pty` 1.0.7** (NuGet, 2026-01-22). ConPTY sur Windows 10 1809+ (via la dépendance
  transitive `Vanara.PInvoke.Kernel32` — pas de natif `win-*` dans le paquet) ; shim natif
  `forkpty`/`openpty` par RID pour `linux-x64/arm64` et `osx-x64/arm64`. API réelle (vérifiée par
  décompilation) = exactement celle supposée au plan : `PtyProvider.SpawnAsync(PtyOptions, ct) →
  IPtyConnection` (`ReaderStream`/`WriterStream` `Stream`, `Resize(cols,rows)`, `Kill()`, `Pid`,
  `ExitCode`, `event ProcessExited`, `IDisposable` **idempotent**). Cible `netstandard2.0` →
  **restaure proprement sur net10 sans NU1701** (asset de première classe), aucune neutralisation.
  - **Pourquoi pas `Pty.Net`** (le nom cité dans l'issue) : abandonné (`microsoft/vs-pty.net`, NuGet
    `0.1.6-pre` de 2018, délisté). Ne pas l'utiliser.
- **Terminal web = `@xterm/xterm` 6.0.0 + `@xterm/addon-fit` 0.11.0**, **vendorisés en UMD** sous
  `src/Piscine.Components/wwwroot/lib/xterm/` (servis à `_content/Piscine.Components/lib/xterm/...`).
  - 6.0 livre UMD **et** ESM, tous deux bundlés en **fichier unique** (0 import de chunk relatif). UMD
    choisi : un fichier par paquet, pas de résolution de chemins de chunks sous `_content/`, chargé par
    injection `<script>` dans le module ESM colocalisé `Terminal.razor.js`.
  - **Piège API** : le wrapper UMD d'addon-fit expose `window.FitAddon` = **la classe elle-même**, pas
    `window.FitAddon.FitAddon` (supposé au plan). `Terminal.razor.js` a un `resolveFitAddonCtor()` qui
    gère les deux formes.

## Repli documenté (si un OS échoue le smoke)

Conforme à la spec §5 (écarté du spike mais prêt) : remplacer le terminal embarqué par un bouton
**« Ouvrir un terminal OS ici »** (`Process.Start` du shell par défaut dans le dossier du dépôt) +
coaching basé sur l'**état du dépôt** (S3, agnostique au shell). Le `PtyService` et le composant
`Terminal` restent en place pour les OS où le PTY marche ; le repli ne s'active que par OS défaillant.

## Limites connues (notées en revue de sprint, acceptables pour un spike)

- **Fenêtre d'orphelin au drop de circuit (hôte SignalR/DevHost)** : `DisposeAsync` du composant tue le
  shell de façon fiable à la navigation et à l'éviction de circuit (défauts `DisconnectedCircuitMax*`
  ≈ 3 min). Mais un kill brutal du process serveur, ou une pression de rétention de circuits, peut
  laisser un shell survivre plus longtemps. **Inhérent à l'hôte SignalR** → **argument fort pour l'hôte
  Photino** (pas de circuit : modèle d'exécution local in-process). Ne s'applique PAS à Photino.
- **Backpressure non gérée** : pas de coalescence dans `OnPtyOutput` — une commande très verbeuse
  (`dir /s`, `ls -R`) peut saturer le circuit SignalR. OK pour le spike (`echo`) ; coalescer
  (buffer + flush ~30 ms) en suivi si besoin. Hôte cible réel = Photino → risque relativisé.
- **`Exited` câblé** (corrigé en revue) : un shell qui se termine (`exit`) affiche désormais
  `[processus termine : code N]` dans xterm au lieu de paraître figé.
- **Sécurité** : `/terminal` = shell OS **sans authentification**. Garde ajoutée : spawn **uniquement en
  Development** (DevHost est hors release) ; règle d'exploitation : **ne jamais binder hors loopback**.

## Pièges réutilisables (pour le HANDOFF / sprints suivants)

- Une page routable `@page` dans la RCL avec interactivité doit déclarer `@rendermode InteractiveServer`
  (le `_Imports.razor` de la RCL importe désormais `@using static …Web.RenderMode`).
- Une page RCL qui consomme un service de `Piscine.App` exige le `ProjectReference`
  `Piscine.Components → Piscine.App` (sens OK, **pas de cycle** : App n'a aucune dép vers Components).
- Vendoriser une lib JS dans une RCL : préférer le **build UMD fichier-unique** (chargé par `<script>`),
  servi sous `_content/<RCL>/...` ; vérifier la **forme réelle du global** (cf. piège FitAddon).
- Le smoke Playwright qui lance le DevHost via `dotnet run` hérite de `ASPNETCORE_ENVIRONMENT=Development`
  du `launchSettings` (profil `http`) → la garde dev-only laisse le terminal s'activer en test.
